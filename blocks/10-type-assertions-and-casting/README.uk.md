# Блок 10 — Твердження типів і приведення

[English](./README.md) · **Українська**

> ℹ️ Джерело істини — англійська версія ([README.md](./README.md)); український переклад може незначно відставати.

Твердження (assertion) каже компілятору: «повір мені, це значення має тип T». Воно змінює
типи **лише на етапі компіляції** — жодної перевірки під час виконання. Ця сила діє в
обидва боки.

## Твердження `as`

```ts
const el = document.getElementById("app") as HTMLDivElement;
```

`as` — **неперевірене**: якщо помилишся, отримаєш баг під час виконання, який компілятор
не зловить. Використовуй лише тоді, коли справді знаєш більше за компілятор — найкраще
одразу після перевірки під час виконання, що доводить тип:

```ts
const data: unknown = JSON.parse(input);
if (typeof data === "object" && data !== null && "x" in data) {
  // безпечно: ми підтвердили форму перед твердженням
}
```

## Твердження non-null `!`

`x!` прибирає `null`/`undefined` з типу `x`. Те саме — неперевірене:

```ts
const first = items[0]!; // «я знаю, що цей індекс існує»
```

## Подвійні твердження

Коли два типи не перетинаються, TS блокує пряме `as`. `x as unknown as T` примушує його —
це «запах коду» (code smell), що майже завжди означає: краще валідуй замість цього.

## `as const`

Стверджує *найвужчі* літеральні типи й робить усе `readonly` (див. блок 05).

## `satisfies`

`satisfies` перевіряє значення на відповідність типу **без його розширення** (widening).
Ти отримуєш і валідацію, *і* точний виведений тип:

```ts
type Config = { host: string; port: number };

const config = {
  host: "localhost",
  port: 8080,
} satisfies Config;
// config.port лишається вузьким типом, а одруківку в ключі буде зловлено.
```

Правило великого пальця: для літералів об'єктів надавай перевагу `satisfies` над `as` —
воно валідує, а не перевизначає.
